# Critical evaluation: ClickUp and Figma

- These tools improved communication and made responsibilities easier to track.
- Figma worked well as a collaborative whiteboard, but only if people kept it updated.
- ClickUp gave the project a more professional production structure, but some team members needed repeated reminders to use it properly.
- This showed that production software is only effective when the team actively engages with it.

---

<!-- footer: (Epic Games, s.d.) -->

![bg right:48% width:100%](https://raw.githubusercontent.com/BradleyCurtisDev/ToolsAndProduction/refs/heads/main/Week%206/scan_results.png)

# Asset scanner tool

- I built a Python script that runs inside the **UE5 editor** using the Python scripting plugin.
- It scans every Static Mesh and Texture2D in a `/Game/DropOff` folder and flags anything that breaks one of four quality rules.
- Results are exported as a **CSV file** and a **Markdown report** so they are easy to read and share with the team.
- This gave everyone a shared, consistent reference point for quality assurance rather than relying on manual checks.

---
<!-- footer:  -->

![bg right:48% width:100%](https://raw.githubusercontent.com/BradleyCurtisDev/ToolsAndProduction/refs/heads/main/Week%206/scan_results.png)

# Report generator and graphs

- I also built a companion script called `generate_report.py` that runs **outside the editor**.
- It reads the CSV, validates the data, and generates visual graphs using **matplotlib**.
- The graphs are embedded directly into the Markdown report so the team can see the scale of issues at a glance.
- Running the script again after new assets are scanned fully **rebuilds** the report from the latest CSV data.

---

# Reflection and Summary

- The most effective work I did was building systems that the **whole team could use**, such as the asset scanner and the Sphinx documentation.
- Managing the Git repository throughout the project meant nobody lost their work, which was a real concern given how Unreal handles binary assets.
- The multiplayer input issue showed that **understanding the networking model** before building systems would have saved significant time.

## What I learned

- Planning for **replication and networking** from the start is essential in multiplayer development.
- Tooling that produces consistent, repeatable output is more valuable than informal checks.
- Good repository management is a production skill just as much as a technical one.
